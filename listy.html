Listy

Personal Project

About

In this personal project, I designed a mobile application that allows families and roommates to organize and distribute list items among themselves. Functional prototyping is implemented using Javascript framework Framer.js.

Main Take Away

Considering a diverse mapping between the interfaces of a mobile application and designing a clear affordance to the interface elements while keeping overall designs simple.

Why

I live with two other friends and we share our grocery costs. Before any one of us go for a grocery shopping, we have to plan out a shopping list on paper, which requires exchange of calls or text messages to coordinate who buys what. Therefore, I thought of creating a mobile application that provides shared lists of items and a common space for shopping-related communication.


Arranging meals for the week, preparing the house for a birthday party, or planning equipment for a camping trip with friends; these are all instances where Listy can be handy. Common to all these events are the following tasks:

  1. Writing a list of food and/or items that will be needed.
  2. Collecting the items as a group, often dividing who buys what.
  3. Making last minute changes such as adding or removing list items.

Process

After a series of sketching sessions, I came up with Axure wireframes in order to focus on user interface behaviors, interactions, and visual elements throughout different pages. I then tested the design with my roommates, because they are the people that would most likely organize the lists.
With the participants, I first gave them a short introduction of the application - "Listy is a utility application which is designed to allow families and roommates the ability to organize and distribute list items among themselves....". I decided not to give them a demo of the app because I primarily wanted to test the intuitiveness of the navigation systems. Two main tasks given to participants were creating an event and add list items.

Result

There were 3 main problems discovered from the test.

  1. Event list and item list (lists that are part of an event) were confusing.
  2. Navigation system was unintuitive- participants didn't easily recognize the "+" button.
  3. Checked items cluttered the screen.

Based on these main findings, I redesigned the user interface of the application by completely changing the navigation system and list organizations.

Redesigned Elements

  1. The only navigation item is a list of events.
  2. Words are used for buttons instead of icons - eg. "All Events", "add", "Done"
  3. Checked items are greyed so that users can focus more on unchecked items.
  4. Individual event page includes a list of items, whereas in the previous design, clicking an event led users to a different page that has a list of items under that event.
  5. Images are used with event description in order to make an event page feel more personal and enjoyable.

Using Javascript framework framer.js, I created a functional prototype of Listy.

Take Away

    • A consideration of the mapping between interfaces
    • One thing that I found most challenging about designing the app was keeping track of the mapping between interfaces. Though I considered the flow through interfaces before I started wireframing, much more complex connections were made between interfaces after I designed individual pages. I believe it was because there are multiple ways to get to each screen from multiple other screens depending on the context of use.
    • Providing a clear affordance while keeping designs simple
    • It was my first time designing flat mobile interfaces and I found it quite challenging to provide users a clear sense of what interactions are available in the interface. For example, when there are some decorations(gradient fill, drop shadow, etc) to a button, users would try to tap it even though they don't know what it exactly does. However, with the flat design, I needed to design a clear affordance to a button so that users know exactly what It does for them. Therefore I used a text("add") instead of sign("+").

Further Implementation

The "quick and dirty" prototype of Listy can be used for qualitative research because qualitative research helps the development team to understand:

  1. behaviors, attitudes and aptitudes of potential users.
  2. technical, business, and environmental contexts - the domain - of the product to be designed.

Qualitative research can also help the progress of Listy project by:

  1. uniting the team with a common understanding of domain issues and user concerns.
  2. empowering management to make more informed decisions about design issues that would otherwise be based on guesswork or personal preference.

I believe once the design team truly understands users, their needs, and their motivations through qualitative research, the usability testing will become an even more efficient tool for judging the effectiveness of design choices made.
